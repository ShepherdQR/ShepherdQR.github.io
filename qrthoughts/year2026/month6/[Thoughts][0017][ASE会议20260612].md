# ASE会议20260612

## 背景
上午，集团会议，公开。

## 整体感受

- 自动化为主，没有见到智能化的味道
- 人机协同只是【口号】级别，ai/agent占据人的生态位是结构性必然
- 大领导有水平，提到了纵向贯通的制度线


## 个人的学习

### 1.1 组织治理

- 软件/组件包从坚硬/稳定的路径，到柔软/支持变化的路径节点
- agent - harness - loop, 关注边界（权限/责任），工具，上下文
- 组织建设，组织能力：可复用的规范/制度。是纵线。loop是横线。
- 从特定的组织，到可复制
- 闭环：协作/功能/资产运营
- repository only
- 文档体系，文件关联性设计（导航）
- 量化验证/回放质量矩阵/交付证据包
- 工程/研发组织方式，需要agent友好，需要MBD
- 代码，配置，手册，追踪，知识库
- 环境，质量，规模化，知识（可链接），治理
- 每次变动范围需要控制：冻结目标与非目标/验证口径
- 可审查切片内容：goal，架构，tech
- 膨胀-压缩：每个节点会有较多同质agent，需要考虑统合；flow随着ai和工具升级需要整理
- 可复用/可运行资产；可复用的任务能力
- 入口地图：工程记录系统
- repo资产化，组织skill，共享知识底座，平台化治理


### 1.2 可关联性追溯的完整证据包
- 证据包：需求-可验证的任务卡-开发-测试
- 人+AI-需求：人和AI多轮修改，明确可测试点/验收标准，生成说明书/spec；
- AI-开发：测试驱动，先出plan，拆分为可测试的步骤，最小修改/实现原则，约定交付需求追踪矩阵的内容和报告格式
- 人-代码审查：问题，矛盾点，边界条件
- 历史决策沉淀

### 2 人机协同

- 协同：人与机器的参与度占比，人多创造性，ai多规则性
- 瑞士奶酪模型，多层防护
- 研发过程工具化，组织资产结构化，体系流程数字化，质量证据自动化
- 角色-组织资产规则-工具链基础/协同引擎-过程记录-研发流程
- CBB，人工随时插入。
- 逐条确认：符合GJB要求
- AI+ MBD，simulink
- 人调用 - 人机协同 - agent为主

### 3 数字化转型和提效实践

- 组织研发重构，效能评估
- 一句话需求-调研-PRD-生成-部署
- MCP，skill，外部生态打进来
- 管理者洞察
- 打通：跨角色往返，对齐
- 效能看板
- 人与AI的边界
- 3个季度的观测
- 核心，潜力种子，运营动作，黑客松，
- 信息衰减，对齐成本低，链路短
- 人：问题抽象，目标/spce设计（需求），工程判断（架构），技术兜底（复杂系统铰链）
- 工具评价：效率，质量，成本，采纳，生成，存留
- 可比，可追踪，可复盘
- 试点团队/场景，基础度量口径，沉淀案例/skill/规范，复制


### 4 智能体驱动的SE

- gap：能识别，能获得，能补足
- 测试：基于需求/大纲的黑盒测试，基于代码的白盒
- 代码语义，混合语义，需求语义。测试的重点是断言生成
- 分析误报的经验沉淀


### 5 专家观点

- 试点 -> 规模化
- 赋能的角色，关键堵点，E2E
- 软件军队建设，牵动组织变革的先锋队
- 技术/管理方式革新，需要重构流程/体系/组织
- 约束条件：安全，可靠，确定
- 赋能与规范性的顺序问题

### 5 大专家观点

- 软件的核心不变量：高安全本质，嵌入式作为主要承载形式，过程能力（约束）保障结果，形式化验证
- 人的角色分配。agent自进化。生产力水平提高。需求-结果的追溯水平提高。

把握好3个关系：
- 过去-现在-未来：已经积累工具链的生命力，资产规范性是能力短板，试点到应用的赋能点怎么做；
- 软件-系统：软件落实系统。训练ai用已有软件，还是开发ai友好的软件；
- 局部-整体：开发式架构下的局部动能。规范化（主要发力点）和高自由度；
  
几个工作：
- 基础设施建设：抓能力，统一大管
- 研管中心：标准/规范，从上到下干什么；
- 试点规范性要强：全要素（业务/组织/流程/资源），工程化（在组织体系下完成）
